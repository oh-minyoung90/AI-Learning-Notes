# RAG (Retrieval-Augmented Generation)

> 검색(Retrieval)으로 찾은 외부 지식을 LLM의 답변 생성(Generation)에 결합하는 기법.
> 한 줄 요약: **"모델에게 물어보기 전에, 관련 자료를 먼저 찾아서 같이 던져준다."**

---

## 1. RAG란?

RAG는 LLM이 답변을 생성할 때, 모델이 학습한 지식(파라미터에 저장된 지식)에만 의존하지 않고,
**외부 지식 저장소에서 관련 문서를 검색해 프롬프트에 함께 넣어주는** 구조를 말한다.

즉, 질문이 들어오면 다음 순서로 동작한다.

1. 질문과 관련된 문서를 지식 저장소에서 **검색(Retrieval)** 한다.
2. 검색된 문서를 질문과 함께 프롬프트에 넣는다.
3. LLM이 이 문서를 근거로 **답변을 생성(Generation)** 한다.

---

## 2. 왜 RAG가 필요한가?

LLM 단독으로 쓸 때 생기는 한계를 보완하기 위해서다.

| 문제 | RAG 없이 (LLM 단독) | RAG 적용 시 |
|------|--------------------|-------------|
| **최신성** | 학습 시점 이후 정보를 모름 | 최신 문서를 검색해 반영 가능 |
| **환각(Hallucination)** | 모르는 걸 그럴듯하게 지어냄 | 근거 문서 기반이라 사실성↑ |
| **도메인 지식** | 사내 문서·비공개 자료를 모름 | 우리 데이터를 검색해 활용 |
| **출처 추적** | 답변 근거를 알 수 없음 | 어떤 문서에서 나왔는지 표시 가능 |
| **비용** | 재학습(fine-tuning) 필요 | 문서만 갱신하면 됨 |

핵심 장점은 **"모델을 다시 학습시키지 않고도, 지식만 갈아끼울 수 있다"** 는 점이다.

---

## 3. 구성요소

RAG 파이프라인은 크게 **① 사전 준비(인덱싱)** 와 **② 실제 질의응답(검색+생성)** 두 단계로 나뉜다.
인덱싱은 미리 한 번(또는 문서가 바뀔 때마다) 해두는 작업이고, 질의응답은 사용자가 질문할 때마다 실시간으로 도는 작업이다.

### 3.1 인덱싱 단계 (사전 준비, offline)

원본 문서를 검색 가능한 형태로 가공해서 저장해두는 과정.

- **Document Loader (문서 로더)**: 원본 데이터를 불러온다.
  - 다양한 소스 지원: PDF, Word, HTML, Markdown, DB, 노션, 구글 드라이브, API 등.
  - 이 단계에서 표·이미지·헤더 같은 요소를 어떻게 텍스트로 뽑아낼지(파싱)가 이후 품질을 좌우한다.
- **Chunking (분할)**: 긴 문서를 검색·처리하기 좋은 작은 조각(청크)으로 나눈다.
  - 왜 나누나? ① 문서 하나가 너무 길면 프롬프트에 다 못 넣고, ② 질문과 관련된 "일부분"만 정확히 찾기 위해서다.
  - 분할 방식: 고정 길이, 문장/문단 단위, 제목 등 구조 기반, 의미 기반(semantic) 등.
- **Embedding (임베딩)**: 각 청크를 의미를 담은 벡터(숫자 배열)로 변환한다.
  - 임베딩 모델 선택이 중요하다 (예: OpenAI `text-embedding-3`, `bge`, `e5`, 한국어면 `ko-sbert` 계열 등).
  - 질문도 나중에 **같은 모델**로 임베딩해야 비교가 성립한다.
- **Vector Store (벡터 DB)**: 임베딩 벡터를 저장하고, 유사도 기반 검색을 빠르게 해준다.
  - 내부적으로 ANN(근사 최근접 이웃) 알고리즘으로 대량 벡터에서 빠르게 유사한 것을 찾는다.
  - 종류: FAISS(로컬/경량), Chroma(로컬), Pinecone·Weaviate(관리형), pgvector(PostgreSQL 확장) 등.
  - 벡터와 함께 **메타데이터**(출처, 날짜, 작성자, 페이지 등)도 저장해두면 나중에 필터링과 출처 표시에 쓸 수 있다.

### 3.2 검색·생성 단계 (실제 질의, online)

- **Query Embedding (질문 임베딩)**: 사용자 질문을 인덱싱 때와 같은 모델로 벡터화한다.
- **Retriever (검색기)**: 질문 벡터와 가까운 청크를 벡터 DB에서 찾는다 (Top-K개).
  - 여기서 검색 방식(벡터/키워드/하이브리드)과 K값을 조정한다.
- **(선택) Reranker (재정렬기)**: 1차로 찾아온 후보들을 더 정밀한 모델로 다시 순위 매김.
- **Prompt Builder (프롬프트 조립)**: 검색된 청크 + 사용자 질문 + 지시문을 하나의 입력 텍스트로 합친다.
  - 이 "프롬프트"가 무엇인지는 아래 [5.4 Prompt / Context 조립](#54-prompt--context-조립) 참고.
- **LLM (생성기)**: 조립된 프롬프트를 받아, 문서를 근거로 최종 답변을 생성한다.
- **Post-processing (후처리)**: 출처(citation) 표시, 형식 정리, 안전성 검사 등을 덧붙인다.

---

## 4. 동작 흐름

```
[인덱싱 단계 - 사전 준비]
원본 문서 → 청킹 → 임베딩 → 벡터 DB에 저장

[질의응답 단계 - 실시간]
사용자 질문
   ↓ (질문도 임베딩)
벡터 DB에서 유사한 청크 검색 (Top-K)
   ↓
검색된 청크 + 질문 → 프롬프트 조립
   ↓
LLM이 근거 기반으로 답변 생성
   ↓
사용자에게 답변 (+ 출처)
```

간단히 말하면 **"질문 → 찾기 → 붙이기 → 답하기"** 의 4단계다.

## 5. 핵심 개념 자세히 보기

> 아래 6개가 RAG 품질을 좌우하는 핵심 개념이다.
> 3개(청킹·임베딩·검색)만으로도 큰 그림은 잡히지만, 실무에서 "왜 답이 이상하지?"를 파고들려면
> **프롬프트 조립·벡터 검색·생성** 단계까지 이해하는 게 좋다.

### 5.1 Chunking (청킹)

문서를 어떻게 자르느냐가 검색 품질에 큰 영향을 준다.

- **너무 크게 자르면**: 관련 없는 내용까지 섞여 노이즈가 많아진다.
- **너무 작게 자르면**: 문맥이 끊겨서 의미가 사라진다.
- **Overlap(겹침)**: 청크 경계에서 문맥이 잘리지 않도록, 앞뒤를 조금씩 겹쳐서 나누는 기법.

> 실무 팁: 고정 길이 분할보다 문단·제목 등 **구조 기반 분할**이 대체로 품질이 좋다.

### 5.2 Embedding (임베딩)

텍스트를 벡터로 바꾸는 과정. 의미가 비슷한 문장은 벡터 공간에서 가깝게 위치한다.
그래서 "키워드가 정확히 일치하지 않아도" 의미 기반으로 찾을 수 있다.

- 질문과 문서를 **같은 임베딩 모델**로 변환해야 비교가 유효하다.
- 유사도는 보통 **코사인 유사도(cosine similarity)** 로 계산한다.

### 5.3 Retrieval (검색)

**Top-K**: 유사도가 높은 상위 K개 청크를 가져온다 (예: K=3~5).

검색 방식은 크게 세 가지가 있다.

| 방식 | 원리 | 강점 | 약점 |
|------|------|------|------|
| **Dense Retrieval** | 임베딩(벡터) 기반 의미 검색 | 표현이 달라도 의미로 찾음 | 정확한 키워드·고유명사에 약함 |
| **Sparse Retrieval** | BM25 등 키워드 기반 검색 | 정확한 단어 매칭에 강함 | 동의어·의미 이해 못 함 |
| **Hybrid Search** | 위 둘을 결합 | 의미 + 키워드 모두 커버 | 구성·튜닝이 복잡함 |

### 5.4 Prompt / Context 조립

**프롬프트(Prompt)** 란 LLM에게 건네는 **입력 텍스트 전체**를 말한다.
ChatGPT 입력창에 쳐 넣는 그 글자들이라고 생각하면 된다.

RAG의 핵심은, LLM에게 질문만 던지는 게 아니라
**"검색해온 문서 + 지시사항 + 질문"** 을 하나의 텍스트로 **합쳐서(조립해서)** 넣는다는 점이다.
이때 프롬프트에 들어가는 참고 자료 부분을 **컨텍스트(Context)** 라고 부른다.

예시 (프롬프트 조립 결과):

```
아래 [참고 문서]를 근거로 질문에 답하세요.
문서에 없는 내용은 지어내지 말고 "모른다"고 답하세요.

[참고 문서]              ← 검색해온 청크 (= 컨텍스트)
1. 연차는 입사 1년 후 15일이 부여된다...
2. 3년 근속 시마다 1일씩 가산된다...

[질문]                   ← 사용자의 원래 질문
우리 회사 연차 규정이 어떻게 되나요?
```

이 전체 덩어리가 "프롬프트"이고, 질문만 있던 것에 참고 문서와 지시문을 끼워 넣는 과정이 **"프롬프트 조립"** 이다.
LLM은 이 컨텍스트를 근거로 답하기 때문에, 학습하지 않은 사내 규정도 답할 수 있게 된다.

- 컨텍스트 길이에는 한계(토큰 제한)가 있어, 무한정 많은 문서를 넣을 수는 없다.
- 지시문(instruction)을 어떻게 쓰느냐에 따라 답변 태도가 달라진다 (예: "모르면 모른다고 답하라").

### 5.5 Vector Store & 유사도 검색

임베딩된 벡터를 저장하고, 질문 벡터와 가까운 것을 빠르게 찾아주는 저장소.

- 수백만 개 벡터에서 일일이 거리를 다 계산하면 느리므로,
  **ANN(Approximate Nearest Neighbor, 근사 최근접 이웃)** 알고리즘으로 "충분히 가까운" 것을 빠르게 찾는다.
- 정확도와 속도는 트레이드오프 관계다 (더 정확히 찾을수록 느려짐).
- 메타데이터를 함께 저장하면 **필터링 검색**(예: "2024년 문서 중에서만")과 **출처 표시**가 가능해진다.

### 5.6 Generation (생성)

검색해온 컨텍스트를 근거로 LLM이 최종 답을 만드는 단계.

- LLM은 컨텍스트를 "참고 자료"로 활용하지만, 반드시 그대로 따르는 건 아니다.
  → 문서와 다른 답을 하거나(환각), 문서를 무시할 수도 있다.
- **Grounding(근거 기반)**: 답변이 실제 컨텍스트에 얼마나 충실한가가 품질의 핵심.
- 좋은 프롬프트 지시문 + 좋은 검색이 받쳐줘야 생성 품질이 올라간다.

## 6. 품질을 높이는 기법

기본 RAG로 부족할 때 자주 쓰는 개선 방법들.

| 기법 | 무엇을 하나 | 언제 유용한가 |
|------|-------------|---------------|
| **Reranking (재정렬)** | 1차로 넉넉히 검색한 뒤, 별도 모델(Cross-Encoder 등)로 정밀 재순위 | 검색은 되는데 순위가 애매할 때 |
| **Query Rewriting (질의 재작성)** | 애매한 질문을 검색에 유리하게 다듬거나 확장 | 사용자 질문이 짧거나 모호할 때 |
| **Hybrid Search** | 벡터 검색 + 키워드 검색 병행 | 고유명사·전문용어가 자주 나올 때 |
| **Metadata Filtering** | 날짜·카테고리·작성자 등으로 검색 범위 제한 | 문서가 많고 조건이 명확할 때 |
| **Contextual Chunking** | 청크에 문서 요약·제목 등 맥락 정보를 덧붙여 저장 | 청크만으론 문맥이 부족할 때 |

## 7. 장점과 한계

### 장점
- 재학습 없이 지식 갱신 가능 (문서만 교체).
- 환각 감소, 사실성·최신성 향상.
- 답변 출처 제시 가능 → 신뢰성·검증 용이.
- 비공개/사내 데이터 활용 가능.

### 한계
- **검색이 틀리면 답변도 틀린다** (Garbage In, Garbage Out).
- 청킹·임베딩·검색 등 튜닝할 요소가 많다.
- 검색 단계가 추가되어 **응답 지연(latency)** 이 늘어난다.
- 프롬프트에 문서를 넣으므로 **컨텍스트 길이·토큰 비용** 제약이 있다.
- 여러 문서에 걸친 복합 추론(multi-hop)에는 여전히 약할 수 있다.

---

## 8. RAG vs Fine-tuning

> **왜 이 비교가 필요한가?**
> RAG를 배울 때 가장 흔한 질문이 *"그냥 우리 데이터로 모델을 다시 학습(fine-tuning)시키면 안 되나?"* 다.
> 둘 다 "LLM을 우리 상황에 맞게 쓴다"는 점이 겹쳐 보여 헷갈리기 쉽다.
> 이 비교의 목적은 **언제 무엇을 쓸지 판단 기준을 세우고, RAG의 정체성(=지식 주입 도구)을 분명히** 하는 것이다.

| 구분 | RAG | Fine-tuning |
|------|-----|-------------|
| 목적 | 지식(정보)을 주입 | 행동·형식·말투를 학습 |
| 갱신 | 문서만 교체하면 됨 | 재학습 필요 |
| 비용 | 상대적으로 저렴 | 학습 비용 높음 |
| 출처 | 제시 가능 | 불가능 |
| 적합한 경우 | 자주 바뀌는 지식, 사내 문서 | 특정 톤·포맷·전문 태스크 |

> 둘은 배타적이지 않다. **"지식은 RAG로, 스타일은 Fine-tuning으로"** 조합해서 쓰기도 한다.

## 9. 평가 (Evaluation)

RAG는 파이프라인이 **검색 → 생성** 두 단계로 나뉘므로, 평가도 나눠서 봐야 한다.
"답이 이상하다"고 할 때, **검색이 문제인지 / 생성이 문제인지**를 구분하는 게 핵심이다.

### 9.1 검색(Retrieval) 평가 — "관련 문서를 잘 찾아왔는가?"

질문에 대한 정답 문서(정답 셋)를 미리 만들어두고, 검색기가 그걸 잘 찾는지 측정한다.

| 지표 | 의미 | 질문 형태 |
|------|------|-----------|
| **Recall@K** | 정답 문서가 상위 K개 안에 포함된 비율 | "필요한 걸 놓치지 않았나?" |
| **Precision@K** | 가져온 K개 중 실제 관련 있는 것의 비율 | "쓸데없는 걸 많이 가져오진 않았나?" |
| **MRR** (Mean Reciprocal Rank) | 첫 정답이 몇 번째에 나왔는지의 평균 | "정답을 위쪽에 배치했나?" |
| **NDCG** | 순위와 관련도를 함께 고려한 정밀 지표 | "순서까지 잘 매겼나?" |

> 검색 지표가 낮으면 → 청킹 방식, 임베딩 모델, K값, 하이브리드 검색 등을 손봐야 한다.

### 9.2 생성(Generation) 평가 — "찾아온 문서로 제대로 답했는가?"

검색은 잘 됐다는 전제하에, LLM이 그 근거로 정확한 답을 냈는지 본다.

| 지표 | 의미 |
|------|------|
| **Faithfulness (근거 충실도)** | 답변이 컨텍스트에 실제로 근거하는가? (환각 여부) |
| **Answer Relevancy (답변 관련성)** | 답변이 질문에 제대로 대응하는가? |
| **Context Precision** | 가져온 컨텍스트 중 답변에 실제 쓰인 비율 |
| **Context Recall** | 정답을 만드는 데 필요한 정보가 컨텍스트에 다 있었는가? |
| **Correctness (정확성)** | 정답(ground truth)과 비교했을 때 사실이 맞는가? |

### 9.3 평가 방법론

- **정답 셋(Ground Truth) 구축**: (질문, 정답, 근거 문서) 세트를 만들어 벤치마크로 삼는다.
- **LLM-as-a-Judge**: 사람 대신 강력한 LLM이 답변 품질을 채점하게 하는 방식. 빠르고 저렴하지만 편향에 주의.
- **사람 평가(Human Eval)**: 가장 신뢰도 높지만 비용·시간이 많이 든다. 최종 검증용으로 활용.
- **오프라인 vs 온라인**: 배포 전엔 벤치마크로(오프라인), 배포 후엔 실제 사용 로그·피드백으로(온라인) 지속 모니터링.

### 9.4 주의점

- 검색 지표가 좋아도 생성이 나쁠 수 있고, 그 반대도 가능하다 → **반드시 나눠서 측정.**
- 자동 지표는 참고용이다. 결국 **실제 사용자 관점의 유용성**이 최종 기준이다.
- 평가 도구: **RAGAS**(RAG 특화 지표), **TruLens**, **DeepEval**, **LangSmith**(추적·평가) 등.

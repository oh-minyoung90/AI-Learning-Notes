# 컨텍스트 엔지니어링 (Context Engineering)

> 모델의 컨텍스트 창에 **무엇을·어떤 형태로·어떤 순서로 넣을지**를 설계·관리하는 기술.
> 한 줄 요약: **"프롬프트가 '어떻게 물을까'라면, 컨텍스트 엔지니어링은 '무엇을 넣을까'다."**

> 관련 노트: 프롬프트 엔지니어링(문장 설계) · RAG(지식 주입) · Harness(에이전트 구조) · LLM(컨텍스트 창).
> 이 문서는 그것들을 **"컨텍스트 창 관리"라는 하나의 관점으로 묶는** 상위 개념이다.

---

## 0. 먼저 알아야 할 기초: 토큰과 컨텍스트 창

본론에 앞서, 이 문서 전체의 바탕이 되는 두 단어부터 잡고 간다.

### 토큰(token)이란?

LLM은 글자를 사람처럼 읽지 않고, 문장을 **작은 조각으로 잘라서** 처리한다. 그 조각 하나가 **토큰**이다.
토큰은 대체로 단어 하나 또는 단어의 일부다.

```
"오늘 날씨가 좋다"  →  "오늘" "날씨" "가" "좋" "다"   (대략 토큰 5개)
```

모델이 다루는 **모든 텍스트는 토큰 단위로 센다.** 입력도 토큰, 출력도 토큰, 비용도 토큰 수 기준.

### "토큰 전부"란?

모델에게 질문할 때 모델이 보는 건 **질문 하나만이 아니다.** 실제로는 아래가 **한꺼번에** 들어간다.

- 시스템 규칙 ("너는 친절한 상담원이다…")
- 지금까지의 대화 이력
- 검색해온 참고 문서 (RAG를 쓴다면)
- 도구 설명 (에이전트라면)
- 그리고 사용자의 질문

이 모두를 토큰으로 바꾼 **총합**이 "토큰 전부"다. 모델은 매번 이 뭉치 전체를 읽고 다음 답을 만든다.

### 컨텍스트 창(context window)이란? — 책상 비유

모델은 이 토큰 뭉치를 무한정 읽을 수 없다. 한 번에 읽을 수 있는 **최대 용량**이 정해져 있고, 그 한계가 **컨텍스트 창**이다.

**책상**을 떠올리면 쉽다. 책상 위에는 종이를 정해진 만큼만 올릴 수 있고, 모델은 **그 위에 놓인 종이만** 읽는다.

```
┌─────────────────────────────────────┐
│  컨텍스트 창 (책상) = 담을 수 있는 최대 토큰   │
│                                     │
│   [시스템 규칙]   "너는 친절한 상담원이다…"   │
│   [대화 이력]     지금까지 주고받은 메시지     │
│   [검색 문서]     RAG로 가져온 참고 자료      │
│   [사용자 질문]   "환불 규정 알려줘"         │
│                                     │
└─────────────────────────────────────┘
        └──▶ 이 안의 토큰 전부를 모델이 읽고 답한다
```

기억할 세 가지:

1. **책상 위(창 안)에 있는 것만 모델이 본다.** 창 밖 정보는 모델에게 없는 것과 같다. "아까 말한 걸 까먹었다"는 건 그 내용이 책상에서 밀려 떨어진 것이다.
2. **책상 크기(창 크기)는 정해져 있다.** 토큰 수로 표현한다(예: 20만 토큰). 꽉 차면 **오래된 것부터 밀려난다.**
3. **입력도 출력도 같은 책상을 쓴다.** 질문·문서·이력이 자리를 차지하고, 모델이 쓰는 답도 같은 공간을 나눠 쓴다.

> 그래서 **컨텍스트 엔지니어링은 결국 "이 책상 위에 무엇을 올리고 무엇을 치울지 정하는 일"** 이다.
> 아래 본문은 이 책상을 어떻게 정리하는지에 대한 이야기다.

## 1. 컨텍스트 엔지니어링이란?

LLM이 답을 생성할 때 실제로 보는 것은 **컨텍스트 창(context window)에 담긴 토큰 전부**다.
사용자 질문뿐 아니라 시스템 프롬프트, 대화 이력, 검색된 문서, 도구 설명, 이전 단계의 결과가 모두 여기 들어간다.

**컨텍스트 엔지니어링 = 이 창에 들어갈 토큰을 목적에 맞게 큐레이션하고 유지·관리하는 일.**

- 2025년 중반, 프롬프트 엔지니어링의 **진화형**으로 등장한 용어다 (Karpathy, Tobi Lütke 등이 대중화).
- 배경: 프롬프트 문구를 다듬는 것만으로는 해결 안 되는 **프로덕션·에이전트 문제**가 드러났기 때문.
- 즉 초점이 **"좋은 문장 찾기"에서 "올바른 정보 구성 찾기"로** 이동했다.

> 핵심 질문의 전환:
> 프롬프트 엔지니어링 → *"이걸 어떻게 표현하지?"*
> 컨텍스트 엔지니어링 → *"모델이 지금 이 단계에서 어떤 정보를 봐야 하지?"*

---

## 2. 프롬프트 엔지니어링과의 차이

둘은 대립이 아니라 **포함 관계**다. 컨텍스트 엔지니어링이 더 넓고, 프롬프트 작성은 그 안의 한 부분이다.

| | 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
|---|---|---|
| 질문 | "어떻게 물을까?" | "무엇을 넣을까?" |
| 대상 | 하나의 지시문(문장) | 창에 들어가는 정보 전체 |
| 시점 | 대체로 1회성 작성 | 매 턴·매 단계마다 반복 큐레이션 |
| 범위 | 프롬프트 | 시스템 프롬프트 + 이력 + 검색 문서 + 도구 + 상태 |
| 적합 | 단일 턴 작업 | 멀티 턴·에이전트·프로덕션 |

> 비유: 프롬프트가 **한 문장**이라면, 컨텍스트 엔지니어링은 **그 문장 주위의 모든 정보 환경을 짜는 일**이다.

---

## 3. 왜 중요한가

### 3.1 컨텍스트 창은 유한한 자원이다

- 창 크기(토큰 수)에는 한계가 있고, 그보다 더 중요하게 **어텐션(집중력)도 유한**하다.
- 정보를 무작정 많이 넣는다고 좋아지지 않는다. 오히려 노이즈가 늘어 **성능이 떨어지고 비용이 증가**한다.

> **OS 비유**: 운영체제가 제한된 RAM에 무엇을 올릴지 관리하듯,
> 컨텍스트 엔지니어링은 모델의 제한된 "작업 기억"에 무엇을 올릴지 관리한다.

### 3.2 에이전트에서 특히 결정적이다

- 챗봇은 한 턴에 필요한 것만 넣으면 된다. 하지만 **에이전트는 루프를 돈다.**
- 47번째 단계에서 판단할 때, 1~46번째 단계의 잔여물이 창에 계속 쌓여 있다.
- 실패의 상당수는 "앞쪽 프롬프트가 나빠서"가 아니라 **"토큰·어텐션 예산을 잘못 써서"** 생긴다.
- → 그래서 Harness(에이전트 구조)와 짝을 이루는 핵심 기술이다.

---

## 4. 컨텍스트를 이루는 것들

창에 들어가는 정보는 성격이 다른 여러 층으로 나뉜다. 사람의 기억에 빗대면 이해가 쉽다.

| 구성요소 | 기억 비유 | 역할 |
|----------|-----------|------|
| **시스템 프롬프트** | 절차 기억 | 행동 규칙·역할·정책 (어떻게 일할지) |
| **대화 이력** | 단기 기억 | 지금까지 오간 메시지 |
| **사용자 선호·프로필** | 일화 기억 | 이 사용자에 대한 지속 정보 |
| **검색된 지식(RAG)** | 의미 기억 | 외부 문서에서 가져온 사실 |
| **도구 정의** | — | 쓸 수 있는 기능과 그 사용법 |
| **상태·이전 결과** | 작업 메모 | 지금까지의 진행·중간 산출 |

> 컨텍스트 엔지니어링은 이 층들을 **매 단계마다 적절히 골라 조합**하는 일이다.

## 5. 핵심 전략 (Write · Select · Compress · Isolate)

LangChain이 정리한 4가지 전략 분류가 널리 쓰인다.

| 전략 | 뜻 | 예시 |
|------|-----|------|
| **Write (쓰기)** | 컨텍스트를 창 밖에 저장해 두었다 필요할 때 꺼냄 | 메모리·스크래치패드에 중간 결과 기록 |
| **Select (선택)** | 지금 필요한 것만 골라 넣음 | RAG로 관련 문서만, 관련 도구만 노출 |
| **Compress (압축)** | 토큰을 줄여 핵심만 남김 | 긴 대화 이력을 요약, 문서 발췌 |
| **Isolate (분리)** | 관련 없는 컨텍스트를 격리 | 하위 작업을 별도 에이전트/창으로 분리 |

- 실전에서는 이 넷을 **조합**한다. 예: 오래된 이력은 요약(Compress)하고, 필요한 문서만 검색(Select)하고, 중간 결과는 메모리에 저장(Write).

---

## 6. 컨텍스트가 망가지는 방식 (실패 모드)

무엇을 넣느냐만큼 **무엇을 안 넣느냐**도 중요하다. 흔히 인용되는 실패 유형들.

| 실패 | 증상 | 원인 |
|------|------|------|
| **Distraction (산만)** | 정보 과다로 핵심을 놓침 | 관련성 낮은 내용까지 다 넣음 |
| **Confusion (혼동)** | 엉뚱한 정보에 휘둘림 | 관련 없는 도구·문서가 섞임 |
| **Clash (충돌)** | 상충되는 정보로 오답 | 모순된 문서·오래된 정보 공존 |
| **Poisoning (오염)** | 잘못된 정보가 계속 재사용됨 | 초기의 오류·환각이 이력에 남아 누적 |

> 원칙: **"많이"가 아니라 "딱 필요한 만큼, 깨끗하게."** 컨텍스트는 채우는 것보다 **덜어내는 것**이 더 어렵고 중요하다.

## 7. 실전 체크리스트

- **관련 없는 건 빼라**: 넣을까 말까 고민되면 대체로 빼는 게 낫다.
- **오래된 이력은 요약하라**: 긴 대화는 원문 대신 핵심 요약으로 대체.
- **필요한 순간에 넣어라**: 모든 걸 처음부터 넣지 말고, 단계별로 그때 필요한 것만.
- **순서를 신경 써라**: 중요한 정보는 눈에 띄는 위치에 (맨 앞/맨 뒤).
- **도구·문서를 좁혀라**: 선택지가 많을수록 모델이 헷갈린다.
- **상태를 창 밖에 두라**: 전부 창에 이고 다니지 말고, 필요할 때 꺼내 쓰는 구조로.
- **오염을 끊어라**: 틀린 중간 결과가 계속 재사용되지 않게 정리·격리.

## 8. 다른 개념과의 관계

컨텍스트 엔지니어링은 기존 개념들을 **"컨텍스트 창에 무엇을 넣는가"라는 한 축**으로 꿰어준다.

| 개념 | 컨텍스트 엔지니어링 관점에서 |
|------|------------------------------|
| **프롬프트 엔지니어링** | 창에 넣는 "지시문"을 잘 쓰는 것 (부분집합) |
| **RAG** | 창에 넣을 "지식"을 검색으로 조달하는 Select 전략 |
| **메모리** | 창 밖에 저장했다 꺼내는 Write 전략 |
| **Harness** | 이 큐레이션을 매 루프마다 자동화하는 구조 |


~~~
┌─ 하네스 (모델을 감싼 시스템 전체) ──────────────┐
│                                              │
│   • 컨텍스트 관리  ← 여기가 "컨텍스트 엔지니어링"     │
│   • 도구 실행                                   │
│   • 검증 / 재시도 루프                            │
│   • 오케스트레이션 (흐름 제어)                      │
│   • 관측 / 로깅                                 │
│   • 안전 / 권한                                 │
│                                              │
└──────────────────────────────────────────────┘
~~~

> 정리: **프롬프트 = 문장, RAG = 지식 조달, 메모리 = 창 밖 저장, 컨텍스트 엔지니어링 = 이 모두를 창 안에 어떻게 배치할지 결정하는 총괄.**

# API (기초 개념부터 AI 모델 API까지)

> **API(Application Programming Interface)** = 프로그램끼리 정해진 규격으로 요청·응답을 주고받게 해주는 "창구".
> 한 줄 요약: **"내부를 몰라도, 약속된 방식으로 요청하면 원하는 결과를 받는다."**

---

## 1. API란 무엇인가?

API는 **프로그램과 프로그램이 대화하는 방법**이다. 내가 만든 프로그램이 다른 시스템의 기능이나 데이터를 쓰고 싶을 때, 그 시스템 내부를 몰라도 **정해진 규격대로 요청**하면 결과를 돌려받는다.

### 식당 비유 (가장 쉬운 그림)

- **손님(내 프로그램)** 은 주방에 직접 들어가지 않는다.
- **메뉴판(API 명세)** 을 보고 주문할 수 있는 것을 안다.
- **웨이터(API)** 에게 주문(요청)을 건네면, 주방(서버·시스템)에서 요리해 음식(응답)을 가져다준다.
- 손님은 주방이 어떻게 돌아가는지 몰라도 된다. **메뉴판에 적힌 방식대로 주문만 하면 된다.**

> 핵심: API는 **"내부 구현을 감추고, 약속된 창구만 노출"** 한다. 그래서 서로 다른 프로그램이 안전하게 협력할 수 있다.

### 왜 API를 쓰나?

- **바퀴를 다시 안 만들어도 됨**: 지도·결제·로그인 같은 걸 직접 구현하지 않고 남의 API를 가져다 쓴다.
- **역할 분리**: 프론트엔드와 백엔드, 서비스와 서비스가 API로 나뉘어 각자 개발.
- **재사용·확장**: 하나의 API를 웹·앱·외부 파트너가 공통으로 사용.

## 2. API가 동작하는 방식 (기본 뼈대)

대부분의 웹 API는 **요청(Request) → 응답(Response)** 구조다. 가장 흔한 REST 방식을 기준으로 본다.

| 구성 | 뜻 | 예 |
|------|-----|-----|
| **Endpoint (URL)** | 어디로 요청하나 | `https://api.weather.com/v1/current` |
| **Method (동작)** | 무엇을 하나 | GET(조회) / POST(생성) / PUT·PATCH(수정) / DELETE(삭제) |
| **Headers** | 부가 정보 | 인증 키, 데이터 형식 |
| **Body** | 보낼 데이터 | 생성·수정 시 넣는 내용(JSON 등) |
| **Response** | 돌아오는 결과 | 상태 코드 + 데이터(JSON) |

### 구체적 예시 — 날씨 조회 API

```
요청:
GET https://api.weather.com/v1/current?city=Seoul
Header: Authorization: Bearer <내-API-키>

응답 (JSON):
{
  "city": "Seoul",
  "temp": 24,
  "condition": "맑음"
}
```

내 프로그램은 날씨를 어떻게 측정하는지 전혀 몰라도, 위 규격대로 요청하면 서울 날씨를 받는다.

### 상태 코드(Status Code) — 요청이 어떻게 됐는지

| 코드 | 의미 |
|------|------|
| **200** | 성공 |
| **201** | 생성됨 |
| **400** | 잘못된 요청 (내가 형식을 틀림) |
| **401 / 403** | 인증 실패 / 권한 없음 |
| **404** | 대상 없음 |
| **429** | 요청 과다 (rate limit 초과) |
| **500** | 서버 오류 (상대 쪽 문제) |

> 앞자리로 기억: **2xx 성공, 4xx 내 잘못, 5xx 서버 잘못.**

### 인증(Authentication)

남의 API를 쓰려면 대개 "내가 누구인지" 증명해야 한다.

- **API 키**: 발급받은 문자열을 요청에 함께 보냄 (가장 흔함).
- **OAuth 토큰**: 사용자를 대신해 접근할 때 (소셜 로그인 등).
- ⚠️ **키는 코드에 하드코딩하지 말고 환경변수·시크릿으로 관리** (8장 참고).

## 3. 개발에서 쓰는 API의 종류

"API"는 하나의 기술이 아니라 여러 형태·용도를 아우른다. 두 축으로 나눠 본다.

### 3.1 통신 방식(스타일)별

| 스타일 | 특징 | 잘 쓰는 곳 |
|--------|------|-----------|
| **REST** | HTTP + URL + JSON. 단순·범용, 가장 흔함 | 대부분의 웹/앱 백엔드 |
| **GraphQL** | 클라이언트가 **필요한 필드만 골라** 요청 | 화면마다 필요한 데이터가 다른 앱 |
| **WebSocket** | 연결을 열어두고 **실시간 양방향** 통신 | 채팅, 알림, 실시간 시세 |
| **gRPC** | 고성능 바이너리(Protobuf), 서비스 간 통신 | 마이크로서비스 내부 통신 |
| **SOAP** | 엄격한 XML 기반(오래된 방식) | 레거시·일부 금융/기업 시스템 |

> REST와 GraphQL은 "데이터를 요청/응답"하는 방식의 차이, WebSocket은 "실시간 연결"이 필요할 때, gRPC는 "서버 간 빠른 통신"에 쓴다고 크게 잡으면 된다.

### 3.2 용도(제공 목적)별 — 개발 시 자주 붙이는 외부 API

| 분야 | 예시 서비스 | 무엇을 대신해주나 |
|------|-------------|-------------------|
| **결제** | Stripe, 토스페이먼츠, PayPal, 아임포트 | 카드 결제·정산 처리 |
| **인증/로그인** | OAuth 기반 구글·카카오·네이버 로그인 | 회원가입·로그인 |
| **지도/위치** | Google Maps, 카카오맵, 네이버 지도 | 지도 표시·길찾기·좌표 변환 |
| **메시징/알림** | 이메일(SendGrid), SMS(Twilio), 푸시(FCM), 슬랙 | 알림 발송 |
| **데이터/공공** | 공공데이터포털, 환율·날씨 API | 외부 데이터 조회 |
| **클라우드/인프라** | AWS·GCP·Azure, 스토리지(S3) | 서버·저장소·배포 |
| **SNS/콘텐츠** | YouTube, X(트위터) API | 콘텐츠 조회·게시 |
| **AI 모델** | OpenAI, Anthropic, Google 등 | 생성·임베딩·이미지 이해 (↓ 4장) |

> 즉 AI 모델 API는 **"용도별 외부 API의 한 종류"** 다. 결제 API로 결제를 대신 처리하듯, AI API로 "지능적 처리"를 대신 맡기는 것.

## 4. AI 모델 API

여기서부터는 위의 "용도별 API" 중 **AI 모델**을 자세히 본다. 호출 구조(요청/응답)는 앞의 일반 API와 똑같고, **다루는 것이 '모델'** 이라는 점만 다르다.

### 4.1 "API로 모델을 쓴다"는 것

| 방식 | 설명 | 특징 |
|------|------|------|
| **자체 호스팅** | 오픈 모델을 내 서버(GPU)에 올려 구동 | 통제·보안 강점, 운영 부담 큼 |
| **API 사용** | 제공사 서버의 모델을 호출 | 인프라 불필요, 최신 모델 즉시 사용, 사용량 과금 |

> AI API도 결국 "호출할 때마다 돈 내는 외부 서비스"다. 종류·비용·지연을 함께 본다.

### 4.2 모델의 종류 (핵심)

"모델"이라고 하면 챗봇 LLM만 떠올리기 쉽지만, API로 제공되는 모델은 **하는 일이 다른** 여러 종류가 있다.

| 종류 | 입력 → 출력 | 무엇에 쓰나 |
|------|-------------|-------------|
| **생성 모델 (LLM/Chat)** | 텍스트 → 텍스트 | 답변·작성·요약·번역·코딩 |
| **임베딩 모델** | 텍스트 → 벡터(숫자 배열) | 의미 검색, RAG, 분류 |
| **리랭커(Reranker)** | (질문, 문서들) → 관련도 순위 | 검색 결과 정밀 재정렬 |
| **멀티모달** | 이미지·음성·텍스트 → 텍스트 | 이미지·문서·화면 이해 |
| **추론(Reasoning) 모델** | 텍스트 → 텍스트(내부 사고 강화) | 수학·논리·복잡한 다단계 |
| **음성(STT/TTS)** | 음성↔텍스트 | 받아쓰기, 음성 합성 |
| **이미지 생성** | 텍스트 → 이미지 | 그림·디자인 생성 |

특히 헷갈리는 셋:

- **생성 vs 임베딩**: 생성 모델은 "답을 쓴다", 임베딩 모델은 "의미를 숫자로 바꾼다". RAG는 **둘 다** 쓴다(임베딩으로 검색, 생성으로 답).
- **리랭커**: 임베딩 검색이 후보를 넓게 뽑으면, 리랭커가 (질문, 문서)를 함께 보고 **정밀 재정렬**. (→ 임베딩/벡터 검색 노트)
- **추론 vs 일반 생성**: 겉은 같은 텍스트 입출력이나, 추론 모델은 답하기 전 **내부적으로 더 오래 "생각"**. 정확하지만 느리고 비싸다. (→ 추론 모델 노트)

### 4.3 호출의 기본 형태 (messages)

대화형 모델은 **역할(role)이 붙은 메시지 목록**을 입력으로 받는다.

```python
# 개념 예시 (제공사마다 필드명은 조금씩 다름)
response = client.chat(
    model="어떤-모델-이름",
    messages=[
        {"role": "system", "content": "너는 친절한 상담원이다."},  # 규칙
        {"role": "user",   "content": "환불 규정 알려줘"},          # 사용자
    ],
    temperature=0.3,
    max_tokens=500,
)
```

- **role**: `system`(규칙), `user`(사용자), `assistant`(모델의 이전 답변).
- 멀티 턴 대화는 이 배열에 이력을 쌓아 보냄 → **컨텍스트 창을 차지**(컨텍스트 엔지니어링 노트).

### 4.4 파라미터 예시

| 파라미터 | 역할 |
|----------|------|
| **model** | 어떤 모델을 쓸지 |
| **temperature** | 무작위성·창의성 (낮으면 일관, 높으면 다양) |
| **max_tokens** | 생성할 최대 토큰 수(출력 길이 상한) |
| **top_p** | 후보 토큰을 누적 확률 상위로 제한 |
| **stream** | 답을 토큰 단위로 실시간 전송할지 |


### 4.5 Function Calling / 구조화 출력

- **Function Calling(도구 사용)**: 모델이 함수·도구를 **호출하겠다고 신호** → 앱이 실행 후 결과를 다시 넣어줌. 에이전트의 기반(Harness 노트).
- **구조화 출력(JSON mode)**: 응답을 JSON 등 정해진 형식으로 강제. 시스템 연동에 필수(프롬프트 노트 3.5).

## 5. AI API의 비용과 모델 선택

- **토큰 단위 과금**: 대개 **입력 토큰 + 출력 토큰** 수로 요금. 출력이 더 비싼 경우가 많다.
- **등급별 가격차**: 크고 똑똑한 모델일수록 비싸고 느리다.
- **프롬프트 캐싱**: 반복되는 앞부분을 캐싱해 비용·지연 절감.
- **모델 라우팅**: 쉬운 작업은 작은 모델, 어려운 작업만 큰 모델로 → 비용 절감.

모델 고르는 기준: 작업 유형(2장) → 필요한 능력 수준 → 컨텍스트 창 크기 → 비용 → 속도 → 멀티모달 여부 → 개방성(폐쇄 vs 오픈) → **데이터 보안(민감 데이터를 외부로 보내도 되나)**.

> ⚠️ 모델 이름·버전·가격은 매우 빠르게 바뀐다. 선택 전 각 제공사 공식 문서 확인. 이 문서는 개념 이해용.

## 6. 자주 쓰는 조합 (다른 노트와 연결)

- **RAG**: 임베딩 모델(검색) + 생성 모델(답변) + (선택) 리랭커 → RAG 노트
- **에이전트**: 생성 모델 + Function Calling + 도구 → Harness 노트
- **문서 처리**: 멀티모달(이미지·스캔 이해) + 생성(요약·추출)
- **비용 최적화**: 모델 라우팅 + 프롬프트 캐싱

## 7. 보안 기본 (모든 API 공통)

- **API 키를 코드에 하드코딩 금지.** 환경변수·시크릿 매니저로 관리, 저장소에 커밋 금지.
- 키 유출 시 **요금 폭탄·데이터 유출** 위험.
- 외부 API로 보내는 데이터의 **민감도·보관 정책**을 확인 (특히 AI API에 사내 데이터 전송 시).
- 요청 제한(**rate limit**, 429)과 오류 처리(재시도·타임아웃)를 설계에 반영.

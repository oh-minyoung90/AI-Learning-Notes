# 프롬프트 엔지니어링 (Prompt Engineering)

> LLM에게 원하는 결과를 얻어내기 위해 **입력(프롬프트)을 설계하는 기술**.
> 한 줄 요약: **"모델을 바꾸지 않고, 지시하는 방식만 바꿔서 결과 품질을 끌어올린다."**

> 관련 노트: LLM(모델 자체) · RAG(지식 주입) · Harness(에이전트 구조). 프롬프트는 이 모든 것의 **가장 기본 입력**이다.

---

## 1. 프롬프트 엔지니어링이란?

**프롬프트(Prompt)** 는 모델에게 건네는 입력 텍스트 전체다. 프롬프트 엔지니어링은 그 입력을
**어떻게 구성하면 모델이 더 정확하고·일관되고·원하는 형식으로 답하는지**를 설계하는 일이다.

핵심 전제는 이것이다.

> 같은 모델이라도 **어떻게 묻느냐에 따라 결과가 크게 달라진다.**
> 성능 문제의 상당수는 "모델이 부족해서"가 아니라 "지시가 모호해서"다.

- 재학습(fine-tuning)이나 추가 비용 없이, **당장 품질을 개선하는 가장 저렴한 수단**이다.
- LLM 활용의 첫 단계이자 기본기다 (LLM 노트의 "Prompting → RAG → Fine-tuning" 순서에서 맨 앞).

---

## 2. 좋은 프롬프트의 구성요소

잘 만든 프롬프트는 대개 아래 요소를 (전부는 아니어도) 명시적으로 담는다.

| 요소 | 역할 | 예시 |
|------|------|------|
| **역할(Role)** | 모델에게 관점·전문성 부여 | "너는 10년차 백엔드 개발자다" |
| **지시(Instruction)** | 무엇을 할지 명확히 | "아래 코드의 보안 취약점을 찾아라" |
| **맥락(Context)** | 판단에 필요한 배경·자료 | "이 서비스는 사내망에서만 동작한다" |
| **예시(Examples)** | 원하는 입출력 형태를 보여줌 | (입력→출력) 쌍 몇 개 |
| **출력 형식(Format)** | 결과 구조 지정 | "JSON으로, 키는 name/risk/fix" |
| **제약(Constraints)** | 하지 말아야 할 것·범위 | "코드 외 설명은 3줄 이내" |

> 전부 넣어야 하는 건 아니다. **작업이 복잡하고 실패가 잦을수록** 이 요소들을 더 명시적으로 채운다.

## 3. 핵심 기법

### 3.1 Zero-shot (예시 없이)

예시 없이 지시만 주는 가장 기본 방식.

```
다음 문장의 감정을 긍정/부정/중립 중 하나로 분류해줘:
"배송은 빨랐지만 포장이 엉망이었다."
```

- 간단한 작업에 충분. 대부분의 시작점.

### 3.2 Few-shot (예시 몇 개 제공)

원하는 입출력 예시를 몇 개 보여줘서 패턴을 학습시킨다 (LLM 노트의 In-context Learning).

```
문장을 감정으로 분류해줘.

입력: "가격도 싸고 배송도 빨라요"  → 출력: 긍정
입력: "두 번 다시 안 삽니다"       → 출력: 부정
입력: "그냥 무난해요"              → 출력: 중립
입력: "배송은 빨랐지만 포장이 엉망" → 출력:
```

- **언제**: 출력 형식을 정확히 고정하고 싶을 때, 미묘한 분류 기준을 예시로 전달하고 싶을 때.
- 보통 2~5개 예시면 충분하고, 예시는 **다양하고 대표성 있게** 고른다.

### 3.3 Chain-of-Thought (CoT, 사고사슬)

"단계별로 생각하라"고 유도해, 추론 과정을 거치게 만든다. 복잡한 계산·논리에서 정확도가 크게 오른다.

```
다음 문제를 단계별로 풀어줘. 각 단계를 설명한 뒤 최종 답을 말해줘.
"사과 3개에 1,200원이다. 사과 7개는 얼마인가?"
```

- **왜 효과적인가**: 모델이 답을 한 번에 내지 않고 중간 과정을 쓰면, 그 과정이 다시 컨텍스트가 되어 더 정확한 결론으로 이어진다.
- **주의**: 최신 추론(reasoning) 특화 모델은 내부적으로 이미 사고를 하므로 CoT를 명시적으로 덜 요구해도 된다.

### 3.4 역할·페르소나 부여 (Role Prompting)

관점과 어조를 설정한다.

```
너는 초등학교 과학 교사다. 아래 개념을 초등학생도 이해하도록 비유를 들어 설명해줘.
```

- 답변의 **전문성·난이도·어조**를 조절하는 데 효과적.

### 3.5 구조화 출력 (Structured Output)

결과를 프로그램이 바로 쓸 수 있는 형식으로 강제한다.

```
아래 이력서에서 정보를 추출해 JSON으로만 출력해줘. 설명은 붙이지 마.
형식: {"name": "", "years": 0, "skills": []}
```

- JSON, 표, XML 태그 등으로 지정. **다른 시스템과 연동할 때 필수** (LangChain의 Output Parser와 연결).
- 팁: "설명 없이 JSON만"처럼 **군더더기 금지**를 명시하면 파싱이 안정된다.

### 3.6 제약·가드 (환각 억제)

```
아래 문서에 근거해서만 답해줘. 문서에 없으면 "자료에 없음"이라고 답해.
```

- RAG와 함께 쓰면 환각을 크게 줄인다 (RAG 노트 참고).

### 3.7 (심화) 그 밖의 기법

| 기법 | 아이디어 | 언제 |
|------|----------|------|
| **Self-Consistency** | 여러 번 추론시켜 다수결로 답 채택 | 정확도가 중요한 추론 |
| **Prompt Chaining** | 작업을 여러 프롬프트로 쪼개 순차 처리 | 한 번에 하기엔 복잡한 작업 |
| **Decomposition** | 큰 문제를 하위 문제로 분해해 지시 | 다단계 과제 |
| **ReAct** | "생각→행동(도구)→관찰" 반복 | 도구를 쓰는 에이전트 (Harness 노트) |

## 4. 실전 작성 원칙

- **구체적으로 지시하라**: "잘 요약해줘" ❌ → "핵심 3가지를 각 1문장으로 요약해줘" ✅
- **긍정형으로 지시하라**: "장황하게 쓰지 마" 보다 "3문장 이내로 써"가 더 잘 통한다.
- **원하는 형식을 보여줘라**: 형식을 말로 설명하는 것보다 **예시 하나**가 강력하다.
- **맥락을 충분히 줘라**: 모델은 네 상황을 모른다. 대상 독자·용도·제약을 명시.
- **한 번에 하나씩**: 지시가 너무 많으면 일부를 놓친다. 복잡하면 나눠라(chaining).
- **모델에게 생각할 여지를 줘라**: 어려운 문제는 "단계별로"를 붙여라(CoT).
- **반복 개선하라**: 프롬프트는 한 번에 완성되지 않는다. 실패 사례를 보고 다듬는다.

### 나쁜 예 → 좋은 예

**나쁜 예**
```
이 글에 대해 알려줘.
```

**좋은 예**
```
너는 기술 편집자다. 아래 블로그 글을 검토해줘.
- 대상 독자: 비개발자 기획자
- 다음 3가지를 각각 2~3문장으로: (1) 핵심 주장 (2) 어려운 용어 (3) 개선 제안
- 전문용어는 쉬운 말로 바꿔 제안해줘.

[글]
...
```

차이: 역할·대상·구체적 지시·출력 형식·제약이 모두 명시됨 → 결과가 안정적이고 재현 가능.

## 5. 안티패턴 (흔한 실수)

| 실수 | 문제 | 대신 |
|------|------|------|
| 모호한 지시 | 매번 다른 결과 | 구체적 기준·수치 제시 |
| 지시 과다 | 일부를 놓침 | 핵심만, 또는 단계 분리 |
| 형식 미지정 | 파싱·연동 곤란 | 출력 형식 예시 제공 |
| 부정형 남발 | 잘 안 지켜짐 | 원하는 것을 긍정형으로 |
| 맥락 생략 | 엉뚱한 가정 | 배경·대상·용도 명시 |
| 한 방에 완성 시도 | 품질 불안정 | 테스트하며 반복 개선 |

## 6. 프롬프트와 모델 파라미터

프롬프트만큼 **샘플링 파라미터**도 결과에 영향을 준다 (LLM 노트 5.2 참고).

- **Temperature 낮게(0~0.3)**: 사실 답변·코드·추출 → 일관적.
- **Temperature 높게(0.8~)**: 브레인스토밍·창작 → 다양함.
- 즉 "정확성이 중요한데 답이 들쭉날쭉"하면, 프롬프트뿐 아니라 temperature도 낮춰본다.

## 7. 기법 선택 가이드

| 상황 | 추천 기법 |
|------|-----------|
| 단순 질의·변환 | Zero-shot |
| 출력 형식을 정확히 고정 | Few-shot + 구조화 출력 |
| 계산·논리·다단계 추론 | Chain-of-Thought |
| 전문성·어조 조절 | 역할 부여 |
| 시스템 연동(JSON 등) | 구조화 출력 |
| 환각 억제 | 제약 + RAG |
| 복잡해서 한 번에 안 됨 | Prompt Chaining / 분해 |
| 도구를 쓰는 자동화 | ReAct (→ Harness) |

---

## 8. 프롬프트도 "평가하고 개선"한다

프롬프트 엔지니어링은 감이 아니라 **반복 실험**이다.

- **테스트 세트**: 대표적인 입력 여러 개로 같은 프롬프트를 돌려본다.
- **비교(A/B)**: 프롬프트 버전별 결과를 비교해 더 나은 쪽을 택한다.
- **실패 수집**: 틀린 사례를 모아 지시·예시를 보강한다.
- 도구: LangSmith 등으로 프롬프트별 결과를 추적·비교하면 편하다 (평가/관측성 주제와 연결).

> 요령: 프롬프트를 바꿀 땐 **한 번에 하나씩** 바꿔야, 무엇이 효과였는지 알 수 있다.

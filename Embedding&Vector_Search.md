# 임베딩 & 벡터 검색 (Embedding & Vector Search)

> 텍스트를 **의미 벡터로 바꾸고(임베딩)**, 그 벡터로 **의미가 가까운 것을 찾는(벡터 검색)** 기술.
> 한 줄 요약: **"단어가 아니라 '의미'로 검색한다."**

> 관련 노트: RAG(전체 파이프라인) · API Model(임베딩·리랭커 모델) · 컨텍스트 엔지니어링(Select).
> RAG 노트가 검색을 개괄한다면, 이 문서는 **검색·임베딩 부분만 깊게** 판다.

---

## 1. 개요

키워드 검색은 "단어가 일치"해야 찾는다. 그래서 "환불"로 검색하면 "반품 규정" 문서를 놓친다.
**벡터 검색은 의미가 가까우면 단어가 달라도 찾는다.** 이걸 가능하게 하는 게 임베딩이다.

두 단계로 이뤄진다.

1. **임베딩(Embedding)**: 텍스트를 의미를 담은 벡터(숫자 배열)로 변환.
2. **벡터 검색(Vector Search)**: 질문 벡터와 가까운 문서 벡터를 찾음.

> **이 문서를 읽는 법 (부담 줄이기)**
> 아래 내용은 깊이를 나눠 읽으면 된다.
> - ✅ **꼭 알기**: 임베딩=의미 좌표 / 인덱싱·검색에 같은 모델 / Dense·Sparse·Hybrid 차이 / 청킹의 중요성 / 리랭커=2단계.
> - 📎 **참고만**: 유사도 수식(2.2), ANN 내부 알고리즘(4장). → 벡터 DB가 알아서 처리하므로 "이런 게 있다" 정도면 충분.

## 2. 임베딩 원리

### 2.1 텍스트 → 벡터, 그리고 "의미 공간"

임베딩 모델은 텍스트를 고정 길이 숫자 배열로 바꾼다. 예를 들어 768차원이면 숫자 768개다.

```
"강아지"  → [0.21, -0.03, 0.88, ... ]   (예: 768개 숫자)
"개"      → [0.20, -0.01, 0.85, ... ]   ← "강아지"와 매우 가까움
"자동차"  → [-0.44, 0.71, 0.02, ... ]   ← 멀리 떨어짐
```

이 숫자들이 놓이는 가상의 공간을 **의미 공간(embedding space)** 이라 한다.
**의미가 비슷한 텍스트는 이 공간에서 가깝게, 다른 텍스트는 멀게** 배치된다.

> **쉽게 말하면**: 모든 단어·문장에 **"의미 좌표"** 를 매기는 것이다. 지도에서 비슷한 동네가 가까이 모여 있듯,
> 의미가 비슷한 말들이 이 좌표 공간에서 서로 가까이 모인다. 검색은 "내 질문 좌표에서 가까운 문서 좌표 찾기"다.

### 2.2 유사도 측정

"가깝다"를 수치로 재는 방법. 주로 셋을 쓴다.

> **쉽게 말하면**: 세 방법 모두 "두 좌표가 얼마나 가까운가"를 점수로 매기는 것이다. 실무에서는 대부분 **코사인 유사도**를 쓰고,
> **1에 가까우면 매우 비슷, 0에 가까우면 무관**이라고만 이해해도 충분하다. 아래 수식은 참고용.

| 방법 | 의미 | 특징 |
|------|------|------|
| **코사인 유사도** | 두 벡터의 **방향**이 얼마나 같은가 (−1~1) | 길이 무시, 가장 널리 쓰임 |
| **내적(dot product)** | 방향 + 크기 함께 반영 | 정규화된 벡터면 코사인과 동일 |
| **유클리드 거리(L2)** | 두 점 사이 **직선 거리** | 값이 작을수록 가까움 |

```python
import numpy as np

def cosine_similarity(a, b):
    a, b = np.array(a), np.array(b)
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# 값이 1에 가까울수록 의미가 유사
sim = cosine_similarity(embed("강아지"), embed("개"))   # 예: 0.94
```

> 팁: 많은 임베딩 모델은 벡터를 **정규화(길이를 1로)** 해서 내놓는다. 이 경우 내적 = 코사인이라 계산이 빨라진다.

### 2.3 임베딩 모델 선택

모델마다 성격이 다르므로 작업에 맞게 고른다.

| 고려 항목 | 내용 |
|-----------|------|
| **차원(dimension)** | 높을수록 표현력↑, 저장·계산 비용↑ (예: 384 / 768 / 1536) |
| **최대 입력 길이** | 한 번에 임베딩할 수 있는 토큰 수. 청크 크기와 맞물림 |
| **언어** | 한국어/다국어 지원 여부 (한국어면 다국어·한국어 특화 모델 확인) |
| **도메인** | 일반 vs 특정 도메인(법률·의료·코드) 특화 |
| **질문-문서 대칭성** | 질문과 문서를 같은 방식으로 임베딩하는지(비대칭 검색용 모델도 있음) |

> **가장 중요한 규칙**: 인덱싱과 검색에 **반드시 같은 임베딩 모델**을 써야 한다. 다른 모델의 벡터는 비교 자체가 무의미하다.

### 2.4 자주 쓰는 임베딩 API (2026 중반 기준)

직접 학습하지 않고 API로 불러 쓰는 대표적인 임베딩 모델들. (이름·가격·버전은 빠르게 바뀌므로 공식 문서 확인)

| 제공 | 대표 모델 | 특징 |
|------|-----------|------|
| **OpenAI** | text-embedding-3-small / -large | 가장 무난한 기본값, 생태계·통합 풍부. large는 차원 축소(MRL) 지원 |
| **Cohere** | embed-v4 | 다국어·멀티모달 강점, 자사 Rerank와 궁합. 한국어 포함 다국어에 좋음 |
| **Voyage AI** | voyage-4 계열 | 검색 품질 상위, 도메인(코드·법률·금융) 특화. 문서/질문 다른 모델로도 같은 공간 사용 |
| **Google** | Gemini Embedding | 멀티모달(텍스트·이미지 등), Google Cloud 생태계 |
| **Jina AI** | jina-embeddings v3/v5 | 긴 문서(대용량 컨텍스트)·다국어, 오픈 가중치 옵션 |
| **오픈/자체호스팅** | BGE-M3, Nomic Embed 등 | 무료·다국어, 직접 호스팅해 비용·보안 통제 |

- **처음엔** OpenAI(무난) 또는 BGE-M3(무료·자체호스팅)로 시작하고, 검색 품질이 병목이면 Voyage·Cohere 등으로 바꿔 비교하는 흐름이 흔하다.
- **한국어 위주**라면 다국어 지원(Cohere·Voyage·BGE-M3 등)을 확인하고, 실제 문서로 Recall을 비교해 고른다.
- **선택 요령**: 벤치마크(MTEB 등)만 믿지 말고, **내 실제 데이터·질문 50~200개로 직접 테스트**해 고르는 게 정석이다.

> ⚠️ 사내·민감 데이터를 외부 임베딩 API로 보낼 때는 보안·규정 검토가 먼저다. 통제가 중요하면 자체호스팅(BGE-M3 등)을 고려.

## 3. 벡터 검색 방식

### 3.1 세 가지 접근

| 방식 | 원리 | 강점 | 약점 |
|------|------|------|------|
| **Dense (밀집)** | 임베딩 벡터 기반 의미 검색 | 표현이 달라도 의미로 찾음 | 정확한 키워드·고유명사에 약함 |
| **Sparse (희소)** | BM25 등 키워드 빈도 기반 | 정확한 단어·희귀어에 강함 | 동의어·의미 이해 못 함 |
| **Hybrid** | 위 둘을 결합 | 의미 + 키워드 모두 커버 | 구성·튜닝 복잡 |

### 3.2 Hybrid 결합 — RRF (Reciprocal Rank Fusion)

Dense와 Sparse 결과를 합칠 때 자주 쓰는 방법. 각 결과의 **순위**를 점수로 바꿔 더한다.

```python
def rrf(rank_lists, k=60):
    # rank_lists: 각 검색기의 [문서id, 문서id, ...] 순위 목록
    scores = {}
    for ranking in rank_lists:
        for rank, doc_id in enumerate(ranking):
            scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
    return sorted(scores, key=scores.get, reverse=True)

# dense와 sparse 결과를 순위 기반으로 융합
final = rrf([dense_results, sparse_results])
```

> 점수 스케일이 서로 다른 두 검색기를 **순위로 통일**해 합치므로 안정적이다.

## 4. ANN — 근사 최근접 이웃 (Approximate Nearest Neighbor)

### 4.1 왜 필요한가

벡터가 수백만 개면, 질문 벡터와 **전부** 거리를 계산(정확한 검색)하기엔 너무 느리다.
그래서 "충분히 가까운" 것을 빠르게 찾는 **근사(approximate)** 알고리즘을 쓴다.

- **정확도 ↔ 속도는 트레이드오프**: 더 정확히 찾을수록 느리고, 빠를수록 일부를 놓칠 수 있다.

> **쉽게 말하면**: 도서관에서 100만 권을 처음부터 끝까지 다 뒤지지 않고, **분류·색인으로 후보 서가만 빠르게 훑는 것**과 같다.
> 아주 가끔 딱 맞는 책을 놓칠 수 있지만, 비교할 수 없이 빠르다. 아래 알고리즘 이름은 **벡터 DB가 내부적으로 처리**하니 "이런 게 있다"만 알면 된다.

### 4.2 대표 알고리즘 (개념)

| 알고리즘 | 아이디어 |
|----------|----------|
| **HNSW** | 벡터들을 계층 그래프로 연결해, 가까운 노드를 따라 빠르게 탐색 (정확도·속도 균형 좋아 널리 쓰임) |
| **IVF** | 벡터 공간을 여러 구역으로 나눠, 질문과 가까운 구역만 탐색 |
| **PQ(양자화)** | 벡터를 압축해 메모리·계산 절약 (약간의 정확도 희생) |

> 대부분의 벡터 DB가 내부적으로 이런 알고리즘을 구현한다. 사용자는 보통 **정확도/속도 파라미터**만 조정한다.

## 5. 리랭커 (2단계 검색)

### 5.1 왜 2단계인가

벡터 검색은 빠르지만 정밀도가 완벽하진 않다. 그래서 흔히 **2단계**로 간다.

1. **1차 검색(Retrieve)**: 벡터 검색으로 후보를 **넓게**(예: 상위 50개) 뽑는다. — 빠름.
2. **2차 재정렬(Rerank)**: 리랭커가 (질문, 각 후보)를 정밀 비교해 순위를 다시 매기고 상위 몇 개만 남긴다. — 정확.

### 5.2 Bi-encoder vs Cross-encoder

| | Bi-encoder (임베딩 검색) | Cross-encoder (리랭커) |
|---|---|---|
| 방식 | 질문·문서를 **따로** 벡터화 후 비교 | 질문·문서를 **함께** 모델에 넣어 관련도 산출 |
| 속도 | 빠름 (벡터 미리 저장 가능) | 느림 (매 쌍마다 계산) |
| 정확도 | 상대적으로 낮음 | 높음 |
| 용도 | 1차 대량 검색 | 소수 후보 정밀 재정렬 |

> 그래서 **"Bi-encoder로 빠르게 좁히고, Cross-encoder로 정밀하게 고른다"** 는 조합이 효과적이다.

## 6. 실무 튜닝

### 6.1 청킹 (Chunking)

임베딩 품질의 절반은 "문서를 어떻게 잘랐나"에서 갈린다.

- **크기**: 너무 크면 노이즈, 너무 작으면 문맥 손실. 임베딩 모델의 최대 입력 길이도 고려.
- **Overlap(겹침)**: 청크 경계에서 문맥이 잘리지 않게 앞뒤를 조금 겹침.
- **구조 기반 분할**: 고정 길이보다 문단·제목 단위가 대체로 낫다.

### 6.2 Top-K 선택

- 너무 적으면 정답 누락, 너무 많으면 노이즈·비용 증가.
- 흔히 1차 검색은 넉넉히(20~50), 리랭크 후 최종은 3~5개로 좁힌다.

### 6.3 메타데이터 필터링

- 날짜·카테고리·작성자 등으로 검색 범위를 먼저 좁히면 정확도·속도가 오른다.
- 인덱싱 때 벡터와 함께 메타데이터를 저장해둬야 가능.

### 6.4 성능·비용

| 항목 | 고려 |
|------|------|
| **인덱싱 비용** | 문서 수 × 임베딩 호출 = 초기 비용. 문서가 바뀔 때만 재인덱싱 |
| **저장 비용** | 차원이 높을수록 저장·메모리↑ (PQ 압축으로 절감 가능) |
| **쿼리 지연** | ANN 파라미터·리랭커 유무가 좌우 |
| **차원 선택** | 높은 차원 = 표현력↑·비용↑. 작업에 과하면 낮은 차원 모델도 충분 |

### 6.5 평가

- **검색 지표**: Recall@K(정답을 상위 K개 안에 넣었나), Precision@K, MRR, NDCG.
- 리랭커·청킹·Top-K를 바꿀 때마다 **같은 테스트셋으로 비교**해야 개선을 확인할 수 있다.
- (자세한 평가 방법은 평가·관측성 노트에서)

## 7. 흔한 실패 / 주의점

- **인덱싱·검색 모델 불일치** → 검색이 완전히 망가진다. (가장 흔한 치명적 실수)
- **청킹이 나쁨** → 아무리 좋은 모델도 관련 없는 조각을 찾아온다.
- **정규화/유사도 방식 혼동** → 잘못된 거리 계산.
- **Top-K를 무작정 키움** → 노이즈와 비용만 증가.
- **의미 검색 맹신** → 정확한 코드·고유명사·숫자는 키워드(Sparse)가 나을 때가 많다 → Hybrid.

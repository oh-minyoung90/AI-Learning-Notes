# MCP (Model Context Protocol)

> AI 모델(LLM)이 외부 도구·데이터·서비스에 **표준화된 방식으로 연결**되게 해주는 개방형 프로토콜.
> 한 줄 요약: **"AI와 도구를 잇는 USB-C 규격."**

> 관련 노트: Harness(도구를 왜·어떻게 쓰나) · API Model(외부 API) · 컨텍스트 엔지니어링.
> Harness 노트가 "도구"를 개념으로 다룬다면, 이 문서는 그 도구를 **어떻게 표준으로 연결하는가**를 다룬다.

---

## 1. MCP란?

**MCP(Model Context Protocol)** 는 LLM이 파일·DB·API 같은 외부 세계와 연결되는 방식을 **하나의 표준으로 통일**한 프로토콜이다.

- Anthropic이 **2024년 11월** 오픈소스로 공개했고, 이후 **Linux Foundation** 산하에서 관리되는 개방형·벤더 중립 표준이 되었다.
- 통신은 **JSON-RPC 2.0** 규격 위에서 이뤄진다.
- 흔한 별명이 **"AI 세계의 USB-C"** 다. 기기마다 다른 충전 단자를 USB-C 하나로 통일했듯, AI-도구 연결을 하나의 규격으로 통일한다는 뜻.

> **"MCP"와 "MCP 서버"는 다른 것이다.**
> - **MCP** = 규칙·표준 자체 (문서로 정의된 약속). 눈에 보이는 프로그램이 아니라 "이렇게 통신하자"는 규격.
> - **MCP 서버** = 그 규칙을 따라 만든 실제 프로그램 (파일 담당, GitHub 담당 등).
>
> 비유하면 **MCP는 "언어(한국어)", MCP 서버는 "그 언어를 말하는 사람"** 이다. 규칙이 있으니, 그 규칙을 따르는 서버라면 어떤 것이든 앱과 통합된다.

---

## 2. 왜 필요한가 — N×M 문제

MCP가 없던 시절엔, **AI 앱마다·도구마다 연결 코드를 따로** 만들어야 했다.

- AI 앱이 N개, 붙일 도구가 M개면 → **N × M 개의 맞춤 연결**이 필요 (조합 폭발).
- 도구가 하나 바뀌면 그걸 쓰는 모든 앱의 연결 코드를 다시 고쳐야 함.

MCP는 이걸 **N + M** 으로 줄인다.

- 각 AI 앱은 MCP **클라이언트**를 한 번만 구현.
- 각 도구는 MCP **서버**를 한 번만 구현.
- 그러면 **아무 앱이나 아무 도구와 자동으로 연동**된다.

> 비유: 나라마다 다른 콘센트에 맞춰 어댑터를 N×M개 만들던 걸, **표준 규격 하나(USB-C)** 로 바꾼 것.

---

## 3. 구조 — 세 가지 역할

MCP는 **클라이언트-서버 구조**이고, 참여자가 셋이다.

| 역할 | 무엇인가 | 예 |
|------|----------|-----|
| **Host (호스트)** | 사용자가 쓰는 AI 애플리케이션 | Claude Desktop, Cursor, VS Code 등 |
| **Client (클라이언트)** | 호스트 안에서 서버 하나와 1:1로 연결되는 세션 | 호스트가 서버 개수만큼 생성 |
| **Server (서버)** | 특정 기능·데이터를 노출하는 작은 프로그램 | 파일시스템, GitHub, DB, Slack 서버 등 |

```
[Host: AI 앱]
   ├── Client A ──▶ [Server: 파일시스템]
   ├── Client B ──▶ [Server: GitHub]
   └── Client C ──▶ [Server: 사내 DB]
```

- 하나의 호스트가 **여러 서버에 동시에** 연결될 수 있다 (각각 별도 클라이언트 세션).
- 서버는 **작고 한 가지에 집중**하도록 설계된다 (파일 담당, DB 담당 식).

> **"클라이언트의 1:1 세션"이란?**
> - **세션(session)** = 한 번 연결해두면 대화 동안 계속 유지되는 통로 (전화 통화처럼, 매번 다시 걸 필요 없음).
> - **1:1** = 클라이언트 하나가 서버 하나만 담당. 호스트가 서버 3개에 연결하면 클라이언트도 3개가 생겨 각각 자기 서버 하나씩만 맡는다.
>
> 비유: 호스트(AI 앱)가 **나**라면, 클라이언트는 내가 **각 상대와 열어둔 개별 전화선**이다.
> 동시에 3통화가 가능하지만(클라이언트 3개), 각 통화선은 정확히 한 사람(서버 하나)하고만 연결돼 있고, 통화가 유지되니(세션) 매번 다시 걸 필요가 없다.

## 4. 서버가 노출하는 것 (3가지 기본 요소)

MCP 서버는 세 종류의 능력을 표준 형식으로 노출한다.

| 요소 | 뜻 | 예 |
|------|-----|-----|
| **Tools (도구)** | 모델이 **실행**할 수 있는 동작(함수) | `search_db(query)`, `send_email(...)` |
| **Resources (리소스)** | 모델이 **읽을** 수 있는 데이터 | 파일 내용, DB 레코드, 문서 |
| **Prompts (프롬프트)** | 재사용 가능한 **프롬프트 템플릿** | "이 코드 리뷰해줘" 같은 정형 지시 |

> 핵심 구분: **Tools = 행동(쓰기·실행), Resources = 읽기(정보 제공), Prompts = 정형 지시.**

> **Tools는 함수인가? 하네스의 Tools와 다른가?**
> - 구현 수준에서 Tools는 **실제로 함수가 맞다.** 도구 하나는 보통 이렇게 생겼다:
>   - 이름(`get_weather`) + 설명("도시명을 받아 날씨 반환" — 모델이 언제 쓸지 판단하는 근거) + 입력(`city`) → 호출되면 코드가 실행돼 결과 반환.
> - **하네스의 Tools와 개념은 똑같다.** 차이는 "제공 방식"뿐이다.
>   - *하네스 노트의 Tools* = "에이전트는 도구를 쓴다"는 일반 개념.
>   - *MCP의 Tools* = 그 도구를 **MCP 표준 형식으로 포장해** 아무 앱이나 쓸 수 있게 노출한 것.
> - 즉 같은 "함수형 도구"이고, MCP는 그걸 **배달하는 표준 규격**이다.

## 5. 통신 방식

- **JSON-RPC 2.0**: 요청·응답을 주고받는 메시지 규격. (어떤 함수를, 어떤 인자로 호출하고, 결과가 무엇인지)
- **전송 방식(Transport)** 두 가지:

| 방식 | 설명 | 쓰임 |
|------|------|------|
| **stdio** | 로컬에서 프로세스 간 직접 통신 | 데스크톱 앱·로컬 개발 |
| **Streamable HTTP** | 원격 서버와 네트워크 통신 | 클라우드 배포·원격 서버 |

## 6. 동작 흐름

```
1. 호스트가 서버에 연결 → 클라이언트 세션 생성
2. 서버가 "내가 제공하는 도구·리소스·프롬프트" 목록을 알려줌 (discovery)
3. 사용자가 질문 → 호스트가 이 목록을 모델에게 알려줌
4. 모델이 "이 도구를 써야겠다"고 판단 → 도구 호출 요청
5. 클라이언트가 서버에 실제 호출 → 서버가 실행하고 결과 반환
6. 결과가 모델의 컨텍스트에 들어가고 → 모델이 최종 답 생성
```

- 즉 모델은 "어떤 도구가 있는지"를 표준 방식으로 발견하고, "그걸 써달라"고 요청만 한다. 실제 실행은 서버가 한다.
- Harness 노트의 에이전트 루프에서 **"도구 실행" 부분을 MCP가 표준화**한 것으로 보면 된다.

## 7. MCP vs Function Calling vs REST API (헷갈리기 쉬운 구분)

셋은 층위가 다르다. 서로 대체가 아니라 **함께** 쓰인다. "서울 날씨 알려줘"를 처리하는 과정을 층별로 따라가면 명확하다.

```
사용자: "서울 날씨 알려줘"
        │
① Function Calling  ─ 모델의 "능력"
   모델: "이건 최신 정보네. 날씨 도구를 써야겠다" → 도구 호출 요청
        │
② MCP  ─ "연결 표준"
   앱이 표준 방식으로 날씨 MCP 서버의 get_weather("서울") 호출
        │
③ REST API  ─ "실제 서비스"
   MCP 날씨 서버가 내부에서 진짜 기상청 REST API 호출
   GET https://api.weather.../current?city=Seoul
        │
   결과("24도, 맑음")가 거꾸로 올라와 → 모델이 최종 답변 생성
```

| 층 | 무엇인가 | 층위 | 비유 |
|---|---|---|---|
| **Function Calling** | 모델이 "도구를 써야겠다"고 판단하는 **능력** | 모델의 기능 | "전화를 걸어야겠다"고 생각 |
| **MCP** | 도구를 앱에 꽂는 **표준 규격** | 연결 규격 | 전화 거는 방식·번호 체계 |
| **REST API** | 실제로 일을 하는 **서비스** | 서비스 인터페이스 | 전화 받은 상대(업무 처리) |

- **MCP는 REST API를 대체하지 않는다.** 오히려 MCP 서버가 내부에서 **REST API를 감싸서** 모델이 쓸 수 있게 만드는 경우가 많다.
- 관계 요약: 모델의 *Function Calling*으로 도구를 호출하고 → 그 도구가 *MCP*라는 표준으로 연결돼 있으며 → MCP 서버는 내부적으로 *REST API*나 DB를 호출한다.

> 한 줄로: **REST/GraphQL은 사람·서비스용, MCP는 그걸 감싸 AI가 쓰게 만드는 표준.**

## 8. 보안 (중요)

도구를 실제로 실행하고 데이터에 접근하므로, 보안이 핵심이다.

- **인증**: 원격 서버는 OAuth 2.1 등으로 권한을 검증.
- **권한 최소화**: 서버가 노출하는 도구·데이터 범위를 필요한 만큼만.
- **위험 행동 통제**: 삭제·전송·결제 같은 되돌릴 수 없는 동작은 승인 게이트를 두는 게 안전 (Harness의 안전 원칙과 연결).
- **신뢰할 수 없는 서버 주의**: 검증되지 않은 MCP 서버는 악의적 도구·데이터를 주입할 수 있으므로 출처 확인 필요.

> 사내 도입 시: 어떤 서버에 어떤 권한을 줄지, 로그·감사를 어떻게 남길지 **처음부터 설계**할 것.

## 9. 생태계·현황 (2026 중반 기준)

- **광범위한 채택**: Anthropic 외 OpenAI, Google, Microsoft, AWS 등 주요 벤더가 지원. 사실상 AI 도구 연동의 표준으로 자리 잡음.
- **공개 서버 다수**: GitHub, Slack, PostgreSQL, Stripe, Figma 등 수백 개의 커뮤니티 서버 존재.
- **거버넌스**: 2025년 12월 Linux Foundation 산하로 이관되어 벤더 중립적으로 관리.
- **발전 중**: 원격 서버 운영, 서버 자동 발견(Server Cards), 장시간 작업(Tasks) 등이 로드맵에 있음.

> ⚠️ 빠르게 발전 중인 표준이다. 스펙 버전·기능은 공식 명세(spec)에서 최신을 확인할 것.

## 10. 다른 노트와의 관계

- **Harness**: MCP는 하네스의 **"도구(Tools)" 구성요소를 표준화**한 것. 도구 실행 층을 재사용 가능하게 만든다.
- **API Model**: MCP 서버는 흔히 외부 REST API를 감싸 모델이 쓰게 해준다.
- **컨텍스트 엔지니어링**: MCP로 가져온 리소스·도구 결과가 컨텍스트 창에 들어간다 → 무엇을 넣을지는 여전히 큐레이션 대상.

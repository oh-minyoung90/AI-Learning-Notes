# Harness Engineering (하네스 엔지니어링)

> 고정된 LLM을 둘러싸고, 그것을 **신뢰할 수 있는 작동 시스템(에이전트)** 으로 만드는 바깥층(scaffold)을 설계하는 분야.
> 한 줄 요약: **"모델이 아니라면, 당신은 하네스다 (If you're not the model, you're the harness)."**

---

## 1. Harness란?

**하네스(harness)** 는 LLM 에이전트에서 **모델 그 자체를 뺀 나머지 전부**를 가리킨다.
즉 프롬프트, 도구(tool) 정의, 메모리, 오케스트레이션 로직, 검증·안전 장치, 로깅 등
**모델을 감싸서 실제 업무를 수행하게 만드는 런타임 인프라**다.

핵심 관점은 이것이다.

> 에이전트 = **모델(지능)** + **하네스(그 지능을 실제로 동작시키는 모든 것)**

- 모델은 "똑똑함"을 제공하지만, 그 똑똑함을 **실행 가능하고·검증 가능하게** 만드는 건 하네스다.
- 하네스는 모델을 더 똑똑하게 만들지 않는다. 대신 모델을 **닫힌 루프(관찰 → 판단 → 행동 → 피드백)** 안의 시스템 일부로 바꾼다.

> RAG도 넓게 보면 하네스의 한 구성요소다 (외부 지식을 컨텍스트로 주입하는 패턴).

### 하네스는 "기능"이 아니라 "구조·설계 개념"이다

자주 하는 오해: 하네스를 켜고 끄는 기능(feature)이나 특정 라이브러리로 생각하는 것.
정확히는 **"고정된 모델을 어떻게 감싸서 신뢰할 수 있게 활용할지"에 대한 설계·엔지니어링 층**이다.

- **모델을 똑똑하게 만드는 게 아니다.** 지능은 그대로 두고, 그 지능을 **안정적으로 꺼내 쓰는 방식**을 설계하는 것이다.
- **추상 개념만도 아니다.** 하네스는 실제로 **편집 가능한 산출물**로 존재한다 —
  프롬프트 템플릿, 도구 명세, 오케스트레이션 코드, 설정 파일, 검증 스크립트, 로깅 훅 등.
- 즉 하네스는 **관점이자 동시에 코드**다. "모델이 아닌 부분은 전부 하네스"라는 사고방식이면서,
  손으로 짜고 고치는 실제 인프라이기도 하다.

> 비유: 모델이 **엔진**이라면, 하네스는 그 엔진을 **자동차로 만드는 나머지 전부**다.
> 차대·변속기·핸들·계기판·브레이크. 엔진 마력(모델 지능)은 그대로여도,
> 나머지를 어떻게 설계하느냐에 따라 완성차의 안전성과 실용성이 완전히 달라진다.

---

## 2. 왜 중요한가?

같은 모델을 쓰더라도, **에이전트의 신뢰성은 모델이 아니라 하네스가 좌우하는 경우가 많다.**

- 실패의 상당수는 모델의 추론 능력 부족이 아니라, **하네스가 정보를 어떻게 노출하고·행동을 어떻게 허용하고·증거를 어떻게 기록하고·정책을 어떻게 강제하느냐**에서 발생한다.
- 긴 호흡의(long-horizon) 작업일수록, 초반의 작은 실수나 누락된 맥락이 **뒤로 갈수록 누적되어** 큰 실패로 이어진다. 하네스가 이를 막는 구조를 제공한다.
- 즉, 하네스는 작업을 모델이 **읽을 수 있고(legible), 실행할 수 있고(executable), 검증할 수 있게(verifiable)** 만든다.

> 실무 격언: *"모델을 바꾸기 전에 하네스부터 고쳐라."* 많은 성능 문제는 모델 교체 없이 하네스 개선으로 해결된다.

## 3. 구성요소

하네스를 이루는 대표적인 요소들. 대부분 **편집 가능한 산출물**(프롬프트 템플릿, 도구 명세, 설정 파일, 검증 스크립트 등)로 구현된다.

| 요소 | 역할 |
|------|------|
| **Prompt / Instruction** | 행동 규칙·태스크 정책·추론 절차를 자연어로 인코딩 |
| **Tools (도구)** | 외부 서비스·기능 노출. 액션 스키마·호출 형식·검증 규칙 정의 |
| **Memory (메모리)** | 대화·작업 이력, 장기 상태를 저장·회수 |
| **Context 관리** | 컨텍스트 윈도우 안에 무엇을 넣을지 선택·압축 (RAG 포함) |
| **Orchestration (오케스트레이션)** | 단계 흐름 제어. 계획→실행→검증 루프, 역할 분담 |
| **Feedback / Verification** | 실행 결과를 검사하고 재시도·수정 유도 (검증 게이트) |
| **Observability (관측성)** | 트레이스·로그로 무엇이 왜 일어났는지 추적 |
| **Safety / Governance** | 권한·접근 제어, 위험 행동 차단, 정책 강제 |

### 구체적으로 어떻게 생겼나 (코딩 에이전트 예시)

추상적으로 느껴질 수 있으니, "버그를 고치는 코딩 에이전트"를 예로 각 요소가 실제로 무엇인지 본다.

| 요소 | 이 에이전트에서의 실제 모습 |
|------|------------------------------|
| **Prompt** | "너는 시니어 개발자다. 테스트가 통과할 때까지 수정하라. 파일 전체를 다시 쓰지 말고 최소 변경만 하라" 같은 지시문 |
| **Tools** | `read_file(경로)`, `edit_file(경로, 변경)`, `run_tests()`, `grep(패턴)` — 각각 입력 형식과 반환 형식이 정의됨 |
| **Memory** | 지금까지 어떤 파일을 봤고 무엇을 수정했는지의 이력, 이전 시도가 왜 실패했는지 |
| **Context 관리** | 저장소 전체(수천 파일)를 다 넣을 수 없으니, 관련 파일·에러 로그만 골라 넣음 |
| **Orchestration** | "코드 읽기 → 수정 → 테스트 → 실패 시 다시 수정"의 반복 루프 제어 |
| **Verification** | `run_tests()` 결과가 통과인지 실패인지 판정 (결정론적) |
| **Observability** | 각 단계에서 무슨 도구를 어떤 인자로 호출했고 결과가 뭐였는지 트레이스 로그 |
| **Safety** | 프로덕션 배포 명령이나 파일 삭제 같은 위험 행동은 차단하거나 사람 승인 요구 |

> 여기서 "모델"이 하는 일은 오직 **"다음에 무엇을 할지 결정"** 하나다.
> 파일을 실제로 읽고·수정하고·테스트를 돌리고·성공을 판정하는 건 전부 하네스가 한다.

## 4. 동작 흐름 (에이전트 루프)

```
목표(Task) 입력
   ↓
[하네스] 프롬프트 + 컨텍스트 + 도구 목록 구성
   ↓
모델(LLM): 다음 행동 결정 (추론 / 도구 호출 / 답변)
   ↓
[하네스] 행동 실행 (도구 호출, 코드 실행, 검색 등)
   ↓
[하네스] 결과 관찰 + 검증 (성공? 오류? 목표 달성?)
   ↓  ── 미완이면 ──┐
   │  결과를 컨텍스트에 반영해 루프 반복
   └───────────────┘
   ↓ (검증 통과 / 종료 조건)
최종 결과 + 트레이스 기록
```

핵심은 **"한 번 물어보고 끝"이 아니라, 관찰–판단–행동–피드백을 반복하는 닫힌 루프**라는 점이다.
LLM은 이 루프 안에서 "다음 한 스텝"을 결정하는 부품으로 동작하고, 나머지 제어는 하네스가 맡는다.

### 실전 예시: 버그 수정 에이전트 한 바퀴

앞의 루프가 실제로 어떻게 도는지, "테스트가 깨진 버그를 고쳐라"는 작업으로 따라가 본다.

| 단계 | 누가 | 무슨 일이 일어나나 |
|------|------|--------------------|
| 1 | 하네스 | 지시문 + 실패한 테스트 로그 + 관련 파일을 프롬프트로 조립 |
| 2 | 모델 | "생각: 로그를 보니 널 체크 누락 같다. 먼저 해당 함수를 읽자" → `read_file` 호출 결정 |
| 3 | 하네스 | 실제로 파일을 읽어 그 내용을 모델에 되돌려줌 |
| 4 | 모델 | "원인 확정. 이 줄에 조건을 추가하자" → `edit_file` 호출 결정 |
| 5 | 하네스 | 파일을 수정하고, 이어서 `run_tests()` 실행 |
| 6 | 하네스 | 테스트 **실패**. 새 에러 로그를 모델에 되먹임 (검증 게이트 미통과) |
| 7 | 모델 | "다른 곳에도 같은 문제가 있었군. 추가 수정하자" → 다시 `edit_file` |
| 8 | 하네스 | 재수정 후 `run_tests()` → **통과**. 루프 종료, 트레이스 기록 |

- 모델은 매 스텝 **"다음에 무엇을 할지"만** 정했을 뿐, 파일 조작·테스트 실행·성공 판정은 전부 하네스가 했다.
- 6번의 **검증 실패 되먹임**이 없었다면, 에이전트는 절반만 고친 채 "완료"라고 답했을 것이다. 이게 하네스가 신뢰성을 만드는 지점이다.

## 5. 핵심 개념 자세히 보기

### 5.1 Agent-Computer Interface (에이전트-컴퓨터 인터페이스)

- 모델이 도구·환경과 상호작용하는 **인터페이스 설계**.
- 같은 모델이라도 **인터페이스를 어떻게 설계하느냐에 따라 성공률이 크게 달라진다** (동일 능력, 다른 결과).
- 예: 도구 출력을 어떻게 보여줄지, 오류 메시지를 어떻게 되돌려줄지, 액션 형식을 어떻게 강제할지.

### 5.2 Planner–Generator–Evaluator (역할 분담 구조)

긴 작업을 안정적으로 처리하기 위해 역할을 나누는 패턴.

| 역할 | 하는 일 |
|------|---------|
| **Planner (계획)** | 목표를 하위 작업으로 분해, 순서 결정 |
| **Generator (실행)** | 실제 행동·산출물 생성 |
| **Evaluator (검증)** | 결과가 맞는지 검사, 통과/재시도 판정 |

- Evaluator를 **LLM으로 둘지, 결정론적 검사(assertion)로 둘지**가 신뢰성에 큰 영향을 준다.
  - LLM 심사: 유연하지만 놓치는 경우 존재.
  - 결정론적 검사(테스트·룰): 엄격하고 재현 가능.

### 5.3 Verification & Feedback Loop (검증·피드백 루프)

- 행동의 결과를 **검사하고, 실패 시 그 정보를 다시 모델에 되먹임**하는 구조.
- "검증 게이트"를 통과해야 다음 단계로 넘어가게 만들면, 오류 누적을 크게 줄일 수 있다.
- 예: 코드 에이전트가 코드를 짜면 → 테스트 실행 → 실패하면 오류를 보고 재수정.

### 5.4 Context Engineering (컨텍스트 설계)

- 제한된 컨텍스트 윈도우에 **무엇을·어떤 형태로 넣을지** 결정하는 것.
- 관련 있는 정보만 골라 넣고, 불필요한 것은 압축·요약·제거.
- RAG, 메모리 회수, 이력 요약 등이 모두 여기에 속한다.

### 5.5 Observability (관측성)

- 에이전트가 **무엇을 왜 했는지** 추적할 수 있게 만드는 것 (트레이스, 로그, 결정 기록).
- 디버깅·평가·개선의 전제 조건. 관측이 안 되면 하네스를 고칠 수도 없다.

### 5.6 Tool Design (도구 설계)

모델이 도구를 잘 쓰느냐는 도구 자체의 성능이 아니라 **"도구를 어떻게 설명하고 노출했느냐"** 에 크게 좌우된다.

- **이름·설명을 명확히**: 모델은 도구 설명만 보고 호출을 판단한다. 모호하면 잘못 쓴다.
- **입력 형식을 단순하게**: 인자가 복잡할수록 실수 확률이 올라간다. 꼭 필요한 것만.
- **에러를 친절하게 되돌려준다**: "잘못됨"이 아니라 "경로가 없음. 유효한 경로 예: ..."처럼 **모델이 스스로 고칠 수 있는 정보**를 준다.
- **출력을 읽기 쉽게**: 원본 그대로보다, 모델이 다음 판단에 쓰기 좋은 형태로 정리해 반환.

> 핵심 통찰: 같은 모델·같은 도구라도, **인터페이스 설계만 바꿔서 성공률이 크게 달라진다.** (→ 5.1 ACI)

### 5.7 대표 실행 패턴 (Agent Loop Patterns)

하네스의 오케스트레이션을 어떻게 짤지에 대한, 자주 쓰이는 정형 패턴들.

| 패턴 | 핵심 아이디어 | 적합한 경우 |
|------|---------------|-------------|
| **ReAct** (Reason + Act) | "생각 → 행동 → 관찰"을 번갈아 반복 | 도구를 써가며 단계적으로 푸는 일반 작업 |
| **Reflexion / Self-Correction** | 실패한 결과를 스스로 되돌아보고 다음 시도에 반영 | 재시도로 품질이 올라가는 작업 |
| **Plan-and-Execute** | 먼저 전체 계획을 세운 뒤, 단계별로 실행 | 단계가 많고 순서가 중요한 긴 작업 |
| **Multi-Agent** | 역할이 다른 여러 에이전트가 협업 (예: 작성자↔검토자) | 관점 분리·상호 검증이 필요한 작업 |

- 대부분의 실전 하네스는 이들을 **조합**한다 (예: Plan-and-Execute 안에서 각 단계는 ReAct로).
- 복잡할수록 좋은 게 아니다. **단순한 패턴으로 되면 단순하게 두는 것**이 유지보수에 유리하다.

## 6. 설계 원칙 (실무 관점)

- **모델은 고정, 하네스로 개선한다**: 재학습보다 하네스 개선이 빠르고 저렴한 경우가 많다.
- **작업을 legible하게**: 모델이 상황을 명확히 이해하도록 정보·도구를 정리해 제시.
- **검증을 내장한다**: "생성 → 검증 → 재시도" 루프를 기본값으로.
- **결정론을 섞는다**: 판단은 모델에, 확인은 결정론적 규칙에 맡겨 재현성 확보.
- **관측 가능하게 만든다**: 모든 단계를 로깅해 실패를 진단 가능하게.
- **작게 시작한다**: 단순 프롬프트 → 도구 추가 → 루프/검증 순으로 점진적으로 키운다.

---

## 7. LLM · RAG · Harness의 관계

| 층위 | 질문 | 초점 |
|------|------|------|
| **LLM** | 모델 자체가 무엇인가? | 토큰 예측, 학습, 능력과 한계 |
| **RAG** | 지식을 어떻게 붙이나? | 검색으로 외부 지식을 컨텍스트에 주입 |
| **Harness** | 모델을 어떻게 시스템으로 만드나? | 도구·루프·검증·관측으로 신뢰성 확보 |

> 세 층은 위로 쌓인다. **LLM(엔진)** 위에 **RAG(지식 공급)** 가 얹히고,
> 그 전체를 **Harness(제어·검증 시스템)** 가 감싸 하나의 에이전트로 완성한다.

## 8. 흔한 실패 요인 (Pitfalls)

- 모델 탓만 하고 하네스를 방치 → 실제 원인은 인터페이스·컨텍스트 문제인 경우가 많다.
- 검증 없는 단방향 생성 → 오류가 조용히 누적된다.
- 관측성 부재 → 왜 실패했는지 알 수 없어 개선 불가.
- 컨텍스트 과다 주입 → 노이즈로 오히려 성능 저하, 비용 증가.
- 도구 스키마·오류 형식이 모호 → 모델이 도구를 잘못 사용.

### "모델 문제"처럼 보이지만 사실 하네스 문제인 경우

에이전트가 실패했을 때, 원인을 모델이 아니라 하네스에서 먼저 찾아야 하는 신호들.

| 증상 | 흔한 진짜 원인 (하네스) |
|------|--------------------------|
| 도구를 자꾸 틀리게 호출한다 | 도구 설명·입력 형식이 모호함 (→ Tool Design) |
| 필요한 정보를 모르는 척한다 | 그 정보가 컨텍스트에 안 들어갔음 (→ Context 관리) |
| 앞에서 한 일을 뒤에서 까먹는다 | 메모리/이력이 컨텍스트에 유지되지 않음 |
| 반쯤 하고 "완료"라고 한다 | 검증 게이트가 없음 (→ Verification) |
| 오류를 반복하며 못 빠져나온다 | 에러 메시지가 고칠 정보를 안 담고 있음 |
| 왜 이렇게 했는지 알 수 없다 | 트레이스·로깅 부재 (→ Observability) |

> 진단 원칙: 실패하면 **"더 좋은 모델"을 찾기 전에, 트레이스를 열어 "하네스의 어느 단계가 부실했나"를 먼저 본다.**
